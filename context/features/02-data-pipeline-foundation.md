# 02 — Data Pipeline Foundation

The build-time GTFS pipeline, minus the frequency math. Downloads TransLink's feed, parses it, emits the static geo/metadata outputs the map will consume. Frequency computation (headways, FTN qualification, per-trip-pattern analysis) lives in `03-frequency-computation` — scoping it out keeps this PR legible.

## Acceptance Criteria

- `npm run prepare-data` runs `scripts/build-data.ts` end-to-end:
  1. Downloads the TransLink GTFS zip to `.cache/gtfs.zip`. Skips the download if the remote file's SHA-256 matches the cached copy.
  2. Parses the relevant files (no full-file loads for anything that might be huge).
  3. Emits three outputs under `public/data/`.
- **`public/data/routes.geojson`** — `FeatureCollection` of `LineString` features, one per shape referenced by at least one trip. Each feature carries `route_id`, `route_short_name`, `route_long_name`, `route_type`, `route_color`, `route_text_color` in its properties. Simplified via mapshaper. **Target: <3 MB gzipped** (per SPEC).
- **`public/data/stops.geojson`** — `FeatureCollection` of `Point` features for every fixed-route stop with `stop_id` and `stop_name` in properties. **Target: <500 KB gzipped.**
- **`public/data/meta.json`** — `{ feed_version, generated_at, translink_attribution, osm_attribution }`. Small, human-readable.
- **HandyDART excluded.** SPEC decision: not fixed-route, not the audience. Filter at the routes level before anything else references route_ids.
- **Streaming parse for `shapes.txt`** (and anything else that could exceed ~20 MB raw). Never `readFile` these wholesale.
- **Unit tests** (Vitest) for the pure helpers: route filtering, shape grouping, property extraction. Use small hand-rolled GTFS fixtures, not the real feed.
- `public/data/` is committed. It's regenerated by the weekly refresh PR (future feature); this PR establishes the initial snapshot.

## Performance / Data Budget

- `routes.geojson` < 3 MB gzipped (after simplification).
- `stops.geojson` < 500 KB gzipped.
- `meta.json` < 2 KB.
- Total data payload stays well under the SPEC's 5 MB budget.

## Out of Scope

- Frequency computation, headway math, FTN qualification → `03-frequency-computation`.
- Parsing of `calendar.txt`, `calendar_dates.txt`, `stop_times.txt` → not needed until frequencies land.
- Weekly GitHub Action for auto-refresh → comes after the pipeline works manually.
- Rendering the outputs on a map → `04-map-skeleton` onward.
- Deduplicating near-identical shape_ids across a route (polish; revisit if the 3 MB budget is tight).

## Notes

- **TransLink GTFS URL:** expected to be `https://gtfs.translink.ca/gtfs/google_transit.zip`. Confirm before committing it as a constant; if wrong, update `scripts/build-data.ts` and re-run.
- **TS execution:** use `tsx` as a dev dep (`npm install -D tsx`) so `npm run prepare-data` → `tsx scripts/build-data.ts` works on both Node 20 and 22 without compiling or relying on experimental flags.
- **CSV parsing:** `csv-parse` (node-csv) has a streaming API (`csv-parse` with `on('readable')`). Prefer it over `papaparse` for streaming.
- **Simplification:** `mapshaper` has a Node API. A command like `mapshaper -i routes.raw.geojson -simplify 10% keep-shapes -o routes.geojson` works. Tune the percentage to hit the budget.
- **Cache key:** SHA-256 of the zip contents, not ETag. TransLink's server may not support conditional GET reliably.
- **Feed version:** prefer `feed_info.txt` → `feed_version` field if present. Fall back to the cached zip's SHA-256 (first 12 chars) so the value is deterministic.
- **Commit plan** (suggested, each buildable on its own):
  - `feat: download and cache gtfs feed with sha256 check`
  - `feat: parse routes and stops from gtfs`
  - `feat: stream-parse shapes and emit routes.geojson`
  - `chore: simplify route geometries with mapshaper`
  - `feat: emit meta.json with feed version and attribution`
  - `test: add unit tests for gtfs parsing helpers`
  - `chore: commit initial public/data snapshot`
